# 状态进度条规则（progress-rule）

WorkSync 每个人卡片下那条横向色块（`buildTimelineBar`）的设计与同步规则。改动进度条前请遵循本文件。

## 1. 它表达什么
- 一条代表**最近 24 小时**的横条，按时间把"这个人处于什么状态"用颜色画出来。
- 数据来源：`rooms/{room}/hist/{uid}/{lid}` = `{ status, ts, message?/enc? }`，每条 = 一次状态变更（`ts` 毫秒时间戳）。
- 颜色 = 状态色（见 `STATUSES`）：工作中绿、专注蓝、开会紫、休息黄、吃饭橙、离开浅灰、下班深灰。

## 2. 视觉规则（当前定稿）
- **24h 刻度**：每段宽 = 该状态持续时长 / 24h × 100%。
- **彩色靠左**：从「最早一条记录」开始，彩色段从**左边缘**依次往右铺（相邻同状态合并成一段）。
- **灰色靠右**：剩下没铺满 24h 的部分用灰色 `#333a47` 补在**右边**，含义 = "没上线 / 没记录"（hover 提示 `tl_norec`）。
- 整条**永远填满**（彩 + 灰），不留透明空白。
- 例：某人 8h 前开始"工作"、没换过 → 左边绿 33.3% + 右边灰 66.7%。
- 说明文案：`📈 近 24h · N 次变化 · 现「X」起于 HH:MM`（`N` = 合并后段数，`X` = 当前状态，时间 = 当前状态起始）。

### 演进史（别走回头路）
踩过的坑，最终选了"彩左+灰右"：
1. 彩色按真实时间定位、当前顶到**右**边缘、左边留灰 → 用户觉得"贴右"。
2. 满铺（从最早记录铺满整条）→ 单一状态变**纯色**（"全绿"），看不出时长。
3. ✅ **彩左 + 灰右（本规则）**：彩色块长度 = 活跃时长占 24h 比例，靠左；右边灰 = 剩余。既不全色也不贴右。

## 3. 数据与同步规则（关键，决定"自己看 == 别人看"）
- **本地优先**：状态一变先写本地 `localStorage.ws_hist`（断网也记），自己卡片的进度条读本地 `loadHist()`。
- **上行 Firebase**：在线时把记录写到自己 hist 节点；`lid` 当子键 → 幂等，重传不产生重复。
- **别人看你**：读 `rooms/{room}/hist/{你的uid}`（Firebase）。所以**本地必须和云端一致**，否则"自己看 12 条、别人看 1 条"。
- **对账（reconcileHist，必须保留）**：仅靠 `synced` 标记补传**不可靠**（标记会被并发写竞态弄错、换 uid、历史失败没复位）。所以连接时 + 每 45s 读一次自己的云端节点，把"本地有、云端缺"的记录**忽略 synced 强制补传**。这是"两人看到一致"的根本保障。
- **清理**：本地 `loadHist` 按 `ts >= now-24h` 过滤；云端 `pruneRemote` 每小时删超 24h 的记录。
- **加密房**：`status` / `ts` 始终明文（进度条只用这两项）；只有 `message` 走 AES-GCM。所以加密与否都不影响进度条。

## 4. 实现要点（`buildTimelineBar(id, isSelf)`）
- `timelineOf`：`isSelf` → `loadHist()`（本地）；否则 → `histById[id]`（来自 `listenHistory` 读的云端，已按 ts 升序）。
- 先 `inWin` = 24h 内、按 ts 升序；空则显示 `tl_none`。
- `pts`：相邻同状态合并（去掉重连产生的冗余 working,working…）。
- 彩色段从左累加 `used`；尾部灰 `tail = 100 - used`。
- `.tl-bar` 是 `display:flex`，段是子元素按顺序排——**所以"靠左/靠右"由段的顺序 + 是否补前导/尾随灰决定**，必须自己算每段 width%。

## 5. 排查"两人不一致"
1. 先确认两边都是**最新版页面**（GitHub Pages 有缓存，让对方 `Ctrl+Shift+R` 强刷）。
2. 仍不一致 → 多半是某人本地有、云端缺：对账会在连接 + 45s 内自动补齐，等几十秒。
3. 要看真实数据：`curl https://<databaseURL>/rooms/{room}/hist.json`（开放规则下 REST 只读，不需 key）对比本地 `ws_hist`。注意用户可能用**自定义房间 / 自己的 Firebase 项目**。
